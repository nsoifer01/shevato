# Testing Steps

## Change 1: Fix firestore.rules syntax error
### What changed
Removed the literal text `Sorry,` that was wedged into the Firestore rules file at line 5, breaking the rules grammar. Without this fix, `firebase deploy --only firestore:rules` would fail and live rules could drift from source.

### Setup
- Firebase CLI installed and authenticated to the `shevato-site` project (`firebase login`).

### Test steps
1. Run `firebase deploy --only firestore:rules --project shevato-site --dry-run` (or, if dry-run unsupported, run `firebase deploy --only firestore:rules`).
2. Verify the deploy completes without a "Compilation error" message.
3. Open the Firebase console → Firestore → Rules and confirm the published rules match `firestore.rules`.
4. Sign in to the gym tracker app at the deployed URL.
5. Verify reading and writing programs/sessions still works (create a program, add a set, finish workout).

### Regression checks
1. Verify rules still deny read/write access to other users' documents — sign in as user A, attempt to read `users/{userB-uid}/apps/gymTrackerApp` via the Firebase console "Rules Playground" and confirm DENY.
2. Verify the granular rule for `users/{uid}/apps/{namespace}` still enforces presence of `data` and `meta.lastUpdated` fields (Rules Playground with a write missing `meta` should DENY).

### Automated checks
- `git diff --check` should pass (no whitespace errors).
- `firebase deploy --only firestore:rules --project shevato-site` should compile and deploy successfully.


## Change 2: Escape user-supplied strings in innerHTML (XSS hardening)

Affects program names/descriptions, workout day names, exercise names, custom exercise fields, achievement icon/name/description, and session notes. The helper already existed (`utils/helpers.js:175`) but was only used in `paused-banner.js`.

2. Open `http://localhost:8000/apps/gym-tracker/`.
3. Click `Programs` → `Create Program`.
4. Enter `Program Name`: `<img src=x onerror=alert('xss-name')>`.
5. Enter `Description`: `<script>alert('xss-desc')</script>`.

## Change 4: Wall-clock-based rest timer
### What changed
`TimerService.startRestTimer` no longer drives its countdown by decrementing a counter inside `setInterval(1000)`. Mobile browsers throttle background tabs and locked-screen tabs, which used to cause real drift — committing a set, locking the phone for 30s, and unlocking would show ~55s remaining instead of ~25s.

The timer now stores `endTime = Date.now() + duration*1000` and recomputes `remaining = ceil((endTime - Date.now())/1000)` on every tick. Tick interval increased to 4 Hz (250 ms) so the visible second-rollover stays smooth. Added `TimerService.extendRestTimer(id, extra)` so `+30s` doesn't have to stop+restart the timer; `workout-view.extendRest` now uses it.

Files touched:
- `apps/gym-tracker/js/services/TimerService.js`
- `apps/gym-tracker/js/views/workout-view.js`

### Setup
- A program with at least one exercise.
- A device or browser tab where you can safely lock the screen / switch tabs.

### Test steps
1. Start a workout and commit a set so the rest bar appears (default 90s).
2. Lock your phone (or switch to a different desktop tab) for 30 seconds.
3. Return to the app. Verify the rest bar shows roughly `0:60` (give or take a second), not `0:90` or some midpoint.
4. Commit another set. Tap `+30s` and verify the value jumps by 30 instantly without the bar refilling to a fresh 100% width — fill should reflect the new remaining/new total.
5. Tap `Skip` mid-rest. Verify the bar disappears.
6. Let a rest timer naturally count down to 0. Verify the bar enters the `Done` state with chime/vibration (per Settings → Alerts).

### Regression checks
1. Foreground rest timer with the tab visible still ticks once per second visually (no flicker, no missed seconds).
2. Workout duration timer (the top header `0:00` clock) is unchanged — it was already wall-clock based.
3. Commit a set, immediately commit the next set before rest ends — the new rest replaces the old one cleanly.

### Automated checks
- `node --check apps/gym-tracker/js/services/TimerService.js` exits 0.
- `node --check apps/gym-tracker/js/views/workout-view.js` exits 0.


## Change 5: Normalize muscle-group casing & strip dead `difficulty` field
### What changed
The exercise database used to mix two `muscleGroup` styles — lowercase-hyphenated (e.g. `'upper-pectorals'`) and Title Case (e.g. `'Upper Chest'`) — and `formatMuscleGroup` papered over both at render time. All Title Case values are now rewritten to their canonical lowercase-hyphenated equivalents. Mappings used:
- `'Chest'` → `'pectorals'`, `'Upper/Lower/Inner/Outer Chest'` → `'upper/lower/inner/outer-pectorals'`
- `'Back'` → `'mid-back'`, `'Lats/Lower Back/Mid Back/Upper Back'` → kebab equivalents
- `'Front/Rear/Side Delts'` → `'front/rear/side-deltoids'`, `'Rotator Cuff'` → `'rotator-cuff'`, `'Shoulders'` → `'shoulders'`
- `'Biceps/Triceps/Forearms/Quadriceps/Hamstrings/Glutes/Calves'` → lowercase
- `'Hip Flexors'` → `'hip-flexors'`, `'Inner Thighs'` → `'adductors'`, `'Tibialis'` → `'tibialis'`
- `'Traps/Neck/Core/Abs/Lower Abs/Obliques/Full Body/Legs'` → lowercase (Lower Abs collapses to abs)

The README claimed the `difficulty` field was removed; it was still present on 278 entries (and unused by any JS). Stripped from every entry.

`formatMuscleGroup` gets one new explicit entry for `'shoulders'` so the Exercises tab subtitle stays predictable.

Files touched:
- `apps/gym-tracker/data/exercises-db.js` (~278 difficulty fields removed, ~30 muscle group values rewritten)
- `apps/gym-tracker/js/utils/helpers.js`

### Setup
- None.

### Test steps
1. Reload the app.
2. Open `Exercises` tab. Filter by category `Chest`. Verify cards show muscle labels like "Chest", "Upper Chest", etc. — same wording as before, no `'pectorals'` raw values leaking.
3. Filter by category `Shoulders` → verify `Shoulders`, `Front Delts`, `Side Delts`, `Rear Delts`, `Rotator Cuff` all render in title case.
4. Open an exercise (e.g. `Barbell Bench Press`). Verify the detail modal still shows muscle group correctly.
5. Filter by `Back` → verify `Lats`, `Mid Back`, `Lower Back`, `Upper Back`, `Outer Lats` are all human-readable.
6. Filter by `Quads` → verify the Quadriceps-mapped exercises now show `Quads`.
7. In the Programs tab, edit any program and verify each exercise row's muscle subtitle is rendered the same as before.

### Regression checks
1. `grep -E "difficulty:" apps/gym-tracker/data/exercises-db.js` returns no matches.
2. `grep -oE "muscleGroup: '[^']+'" apps/gym-tracker/data/exercises-db.js | sort -u` returns only lowercase-hyphenated values.
3. Custom exercises with user-typed muscle groups (free-text) still render via the fallback path (`formatMuscleGroup`'s string-massage at the bottom).
4. Existing user data continues to work — sessions stored under old display names use `exerciseName` (denormalized) so they're unaffected.

### Automated checks
- `node --check apps/gym-tracker/data/exercises-db.js` exits 0.
- `node --check apps/gym-tracker/js/utils/helpers.js` exits 0.


## Change 6: Stop cross-app sync on every page load
### What changed
`sync-system/app-sync-init.js` used to open Firestore listeners and run an initial-merge transaction for every app's namespace (`marioKartApp`, `footballH2HApp`, `gymTrackerApp`) on every app page load, "so they're ready when user navigates." Each app is a separate static HTML page so navigating between them is a hard reload — the warm-cache argument doesn't hold, and the cross-app sync was pure cost.

Now `initAppSync` only opens listeners for (a) the current page's namespace and (b) the shared `globalPrefs` namespace.

Files touched:
- `sync-system/app-sync-init.js`

### Setup
- Sign in to the gym tracker (Firebase Auth via the existing flow).
- Open DevTools → Network tab → filter by `firestore`.

### Test steps
1. Reload `apps/gym-tracker/index.html`.
2. Inspect the WebSocket / `Listen` requests in the Network tab. You should see one Firestore listener path containing `users/{uid}/apps/gymTrackerApp` and one containing `users/{uid}/apps/globalPrefs`.
3. Verify there are NO listeners for `marioKartApp` or `footballH2HApp`.
4. Make a change (start a workout, commit a set) and confirm sync still works — open the same account on another device and verify the change appears within a couple of seconds.
5. Sign out then sign in again — verify there are still only two listeners on the gym page.

### Regression checks
1. Mario Kart page (`apps/mario-kart/`) and Football H2H page each still open exactly their own namespace + globalPrefs.
2. Theme preference still syncs cross-app (because `globalPrefs.theme` is still in the keep list).

### Automated checks
- `node --check sync-system/app-sync-init.js` exits 0.
- `grep -n "appName !== currentApp" sync-system/app-sync-init.js` returns no matches (the cross-app loop has been removed).
